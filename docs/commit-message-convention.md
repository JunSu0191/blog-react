# 커밋 메시지 규칙

## 헤더 형식

커밋 헤더는 아래 형식으로 작성한다.

```txt
feat(domaind): 핵심 요약
fix(domaind): 핵심 요약
refactor(domaind): 핵심 요약
```

- `feat`: 사용자 기능 추가 또는 사용자 가치가 있는 흐름 개선
- `fix`: 버그 수정, 예외 처리 보완, 동작 안정화
- `refactor`: 동작 변화 없이 구조 정리

## domaind 규칙

`domaind` 자리에는 변경이 모이는 도메인 이름을 넣는다.

```txt
chat
comment
post
blog
social
app
shared
docs
branding
```

여러 도메인이 함께 바뀌면 가장 중심이 되는 도메인을 선택한다. 애매하면 `app` 또는 `shared` 대신 실제 사용자 기능 도메인을 우선한다.

## 본문 작성 규칙

본문은 flat bullet 2-4개로 작성한다.

- 각 bullet은 완료된 결과를 적는다.
- `개선`, `처리`, `정의`, `정리`, `분리`, `보완`, `연결` 같은 표현을 사용한다.
- `무엇을 한다`, `처리할 예정`, `지원할 수 있게 함` 같은 미완료 표현은 쓰지 않는다.
- 한 줄 bullet도 결과형 명사 문장으로 끝낸다.

## 좋은 예시

```txt
feat(chat): 모바일 채팅 레이아웃 개선
- 데스크톱/모바일 대화방 렌더링 분리
- 대화방 액션 메뉴 처리
- 입력창 포커스 시 스크롤 정렬 보완
```

```txt
feat(comment): 댓글 상호작용 흐름 개선
- 모바일 댓글 시트 처리
- 댓글 액션 피드백 정리
- 댓글 조회 캐시 갱신 보완
```

```txt
fix(post): 에디터 미디어 블록 저장 처리
- 이미지 노드 직렬화 처리
- 링크 카드 파싱 보완
- 작성 화면 데이터 매핑 정리
```

```txt
fix(docs): 커밋 메시지 규칙 정의
- 헤더 포맷 정의
- 결과형 bullet 규칙 정리
- 금지 표현 기준 보완
```

## 피해야 할 예시

```txt
feat(chat): 채팅 기능 추가
- 채팅 기능을 추가한다
- 모바일 처리를 한다
```

```txt
fix(comment): 댓글 수정
- 버그 수정
```

```txt
fix(post): 에디터 저장 수정
- 저장을 한다
- 미디어를 처리한다
```

위처럼 의미가 넓거나 미완료 표현으로 끝나는 메시지는 사용하지 않는다.
